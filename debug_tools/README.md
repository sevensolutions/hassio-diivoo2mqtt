# DIIVOO Debug Tools

This folder contains experimental tools for debugging and reverse engineering the original DIIVOO gateway protocol.

The tools are meant for development only. They are not required for normal diivoo2mqtt operation.

## Purpose

The normal diivoo2mqtt setup uses a flashed ESP32 gateway to communicate directly with DIIVOO irrigation timers over RF.

These debug tools solve a different problem: they allow observing the original DIIVOO gateway while it is still being used normally by the official DIIVOO app.

This is useful because the original gateway already knows how to talk to the valves. By tapping its serial debug output, we can inspect the real packets exchanged between the original gateway and the valves without needing a second RF gateway to listen on the air.

This helps with:

- Reverse engineering unknown protocol fields
- Comparing original gateway behavior with diivoo2mqtt behavior
- Understanding wake-up, parameter sync, schedule sync, and rain delay packets
- Capturing packets generated by the official DIIVOO app
- Debugging protocol assumptions before implementing them in the ESP32 gateway or backend

## Architecture

```text
Official DIIVOO App
        |
        v
Original Chinese DIIVOO Gateway
        |
        | Serial debug output
        | g_pRxBuffer / g_pTxBuffer
        v
diivoo-virtual-serial-gateway
        |
        | TCP on port 4515
        v
diivoo-bridge
        |
        v
Human-readable decoded packet output
```

## Tool 1: diivoo-virtual-serial-gateway

Path:

```text
debug_tools/diivoo-virtual-serial-gateway/
```

This tool reads the serial debug output of the original DIIVOO gateway.

It watches for debug lines containing:

- `g_pRxBuffer` for packets received by the original gateway
- `g_pTxBuffer` for packets transmitted by the original gateway

It then exposes these packets over TCP in a format that can be consumed by local debugging tools.

Output lines:

```text
ORIG_RX:<hex>
ORIG_TX:<hex>
```

Meaning:

- `ORIG_RX` means valve to original gateway
- `ORIG_TX` means original gateway to valve

The tool also accepts basic ESP32-style commands such as `TUNE:` and `TX:` and can fake ACK responses. This is useful when connecting tools that expect an ESP32-like TCP gateway, even though no RF transmission is performed by this virtual gateway.

## Tool 2: diivoo-bridge

Path:

```text
debug_tools/diivoo-bridge/
```

This tool connects to the virtual serial gateway over TCP and decodes the packets.

It understands:

- `ORIG_RX:<hex>`
- `ORIG_TX:<hex>`
- ESP32-style `RX:` lines

It prints packet fields such as:

- Header
- Target ID
- Sender ID
- Sequence byte
- Command byte
- Payload length
- Decoded payload fields

Payload decoding is modular. Decoders are stored in:

```text
debug_tools/diivoo-bridge/decoders/
```

For example:

- `0x20.js` wake / refresh trigger
- `0x05.js` parameter request
- `0x85.js` parameter response
- `0x06.js` schedule request
- `0x86.js` schedule response
- `0x21.js` valve action command
- `0xA1.js` action acknowledgement

## Typical Workflow

1. Connect a USB-to-serial adapter to the original DIIVOO gateway UART/debug pins, as you would for flashing, but leave the original firmware running.
2. Start the virtual serial gateway.
3. Open the official DIIVOO app and use the original gateway normally.
4. Start `diivoo-bridge`.
5. Watch decoded packets live while the official app talks to the original gateway.

This allows observing the real protocol behavior without replacing the original gateway and without using a second RF receiver.

## Hardware Connection

To use these tools, you need to connect a USB-to-serial adapter to the original DIIVOO gateway.

Use the same kind of serial connection you would use when preparing the gateway for flashing, but do not flash it. The original firmware must keep running so the official DIIVOO app can continue to use the gateway normally.

In other words:

- Open the original gateway.
- Connect your USB-to-serial adapter to the gateway UART/debug pins.
- Keep the original firmware installed.
- Start the virtual serial gateway on your computer.
- Use the official DIIVOO app normally.
- The tools will read the gateway's serial debug output and decode the RF packets.

Make sure the serial adapter voltage level matches the gateway logic level. Do not use 5 V TTL if the gateway uses 3.3 V logic.

## Setup

Install dependencies for the virtual serial gateway:

```sh
cd debug_tools/diivoo-virtual-serial-gateway
npm install
```

Start the virtual serial gateway:

```sh
node virtual_gateway.js
```

By default it uses:

```text
Serial port: /dev/ttyUSB0
Baud rate:   460800
TCP port:    4515
```

Adjust these values in `virtual_gateway.js` if needed.

Start the packet decoder bridge:

```sh
cd debug_tools/diivoo-bridge
node app.js
```

By default it connects to:

```text
127.0.0.1:4515
```

## Notes About Packet Direction

`ORIG_TX` packets are packets sent by the original gateway to the valve.

`ORIG_RX` packets are packets received by the original gateway from the valve.

This is important because some command IDs have direction-specific meaning.

For example:

- `0x20` is usually hub to valve and appears to trigger wake-up / refresh behavior
- `0x05` is valve to hub and requests parameter data
- `0x85` is hub to valve and answers parameter requests
- `0x06` is valve to hub and requests watering schedules
- `0x86` is hub to valve and answers schedule requests

## Reverse Engineering Notes

The current understanding is incomplete and based on observed traffic.

Known or suspected behavior:

- `0x20` is a long-preamble wake / refresh trigger.
- After `0x20`, valves often request parameters using `0x05`.
- The hub responds to `0x05` with `0x85`.
- Rain delay is encoded in the `0x85` payload at bytes `[9..12]` as a Tuya-style 32-bit date.
- Valves request schedules using `0x06`.
- The hub responds with `0x86`.
- Multi-page schedules are handled by follow-up `0x06` requests.

Unknown or not fully confirmed:

- Exact meaning of every `0x20` payload variant
- Whether the second byte of the 2-byte `0x20` payload is a fixed value, subtype, or trigger mode
- Full behavior of possible `0x20` variants such as config-only or schedule-only refresh triggers

## Limitations

These tools do not replace the production ESP32 gateway.

The virtual serial gateway is mostly passive. It observes the original gateway and forwards decoded packets. It does not transmit RF packets itself.

The virtual gateway currently emits `ORIG_RX` and `ORIG_TX` lines, which are intended for `diivoo-bridge`. The production diivoo2mqtt backend expects ESP32-style `RX:` lines and will not treat `ORIG_RX` / `ORIG_TX` as normal radio packets without additional support.

## When To Use This

Use these tools when you want to understand how the official DIIVOO gateway behaves.

Typical examples:

- Capturing what happens when the official app changes rain delay
- Capturing what happens when schedules are added or removed
- Comparing official gateway packets against diivoo2mqtt packets
- Investigating unknown valve models
- Validating protocol assumptions before changing production code

## Safety

These tools are experimental and intended for local development only.

Do not expose the virtual TCP gateway to untrusted networks.
